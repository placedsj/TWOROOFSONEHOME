## 2024-03-18 - Avoid Exposing Server-Side Env Vars to Client via Vite Define
**Vulnerability:** The `vite.config.ts` was using the `define` configuration block to inject `GEMINI_API_KEY` directly into `process.env.API_KEY` and `process.env.GEMINI_API_KEY`. This embedded the API key within the compiled client-side JavaScript bundle, making it accessible in plaintext to anyone examining the source code of the web application.
**Learning:** Vite's `define` option replaces string occurrences globally during the build. Injecting sensitive environment variables here, even if intended for a local development setup or specific use case, critically exposes them to the client. This is a common pattern when developers try to use Node.js `process.env` idioms in Vite apps, rather than relying on Vite's built-in `import.meta.env` system for safe, public client-side variables (`VITE_*`).
**Prevention:** Never use Vite's `define` property to pass sensitive secrets to the client build. All secrets and sensitive operations (such as communicating with external APIs like Gemini) must remain entirely server-side, or if a specific API allows client-side access, only safe, non-privileged, or tightly-scoped, short-lived tokens should be provided. In this repo, simply omitting the `define` configuration prevents the injection, assuming that logic consuming it in this codebase was refactored (or removed, if unnecessary client-side usage).
